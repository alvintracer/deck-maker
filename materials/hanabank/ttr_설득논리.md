## Part 1. 공통 배경: 금융기관은 기존 TR 솔루션을 직접 못 쓴다

### 1.1 금융회사 망분리 규제

금융회사(은행, 증권사, 보험사)는 **전자금융감독규정**에 따라 내부 업무망과 외부 인터넷망을 물리적·논리적으로 분리해야 한다. 이는 단순한 방화벽 수준이 아닌, 업무 서버가 인터넷에 직접 접속할 수 없는 구조다.

```
┌──────────────────────────────────────────┐
│              금융회사 내부                │
│  ┌────────────┐     ┌────────────┐      │
│  │ 업무서버    │  ✕  │ 인터넷     │      │
│  │ (KYC/AML)  │     │ (HTTPS)    │      │
│  └────────────┘     └────────────┘      │
│       │ 내부망           │ DMZ           │
│       │                  │               │
│  ─────┴──── 망분리 ──────┘               │
│                                          │
│  외부 통신은 반드시:                      │
│  ① 전용선 (Leased Line)                 │
│  ② VPN                                  │
│  ③ 금융망 (금결원, VAN)                  │
│  을 통해서만 가능                         │
└──────────────────────────────────────────┘
```

**기존 TR 솔루션(CODEVASP, VerifyVASP, Sumsub, GTR)은 모두 인터넷 기반 HTTPS API로 통신한다.** 금융회사 업무서버에서 이 API를 직접 호출하는 것은 망분리 규정 위반이다.

### 1.2 VerifyVASP의 Enclave Docker — 금융회사에서 불가능한 이유

VerifyVASP는 각 VASP에 **Enclave**라는 Docker 컨테이너를 설치하는 방식이다:

```
┌─ 일반 VASP (거래소) ──────────────────────┐
│                                            │
│  ┌──────────────┐     ┌──────────────┐    │
│  │ 거래소 서버   │◄───►│ VV Enclave   │    │
│  │ (KYC DB)     │     │ (Docker)     │    │
│  └──────────────┘     └──────┬───────┘    │
│                              │ HTTPS       │
│                              ▼             │
│                     VV Central Server      │
│                     (verifyvasp.com)        │
└────────────────────────────────────────────┘
```

**금융회사에서 이 구조가 불가능한 이유:**

| 문제 | 상세 |
| --- | --- |
| **① 제3자 블랙박스 컨테이너** | Enclave는 VerifyVASP가 제공하는 Docker 이미지. 금융회사 보안 정책상, **소스코드가 공개되지 않은 제3자 컨테이너**를 내부 보안 영역에 설치·실행하는 것은 IT보안 감사 및 ISMS 인증 기준에 부합하지 않음 |
| **② 내부망에서 외부 HTTPS 통신** | Enclave가 VV Central Server와 상시 HTTPS 통신을 해야 함. 이는 금융회사 내부 보안 영역에서 **인터넷 아웃바운드 연결**을 여는 것으로, 망분리 원칙 위반 |
| **③ KYC DB 접근 권한** | Enclave가 고객 KYC 데이터(이름, 생년월일 등)에 접근해야 본인 확인이 가능. **제3자 컨테이너에 고객 개인정보 DB 접근 권한을 부여**하는 것은 개인정보보호법 및 내부 정보보호 정책에 따라 심각한 리스크 |
| **④ 컨테이너 업데이트 통제 불가** | VV가 Enclave Docker 이미지를 업데이트하면 금융회사도 따라야 함. **외부 업체가 금융회사 내부 인프라의 소프트웨어 버전을 사실상 통제**하는 구조 |
| **⑤ 장애·사고 책임 소재** | Enclave 장애 시 금융회사 TR 처리가 중단됨. 금융회사 내부에서 돌아가는 제3자 컨테이너의 **장애 대응 주체와 책임 범위**가 불명확 |
| **⑥ 금융감독원 검사 대응** | 금감원 IT검사 시 "왜 제3자 Docker 컨테이너가 내부망에서 인터넷으로 통신하며 고객 개인정보를 처리하는가?"에 대한 합리적 설명이 어려움 |

> **결론: VerifyVASP의 Enclave 방식은 일반 거래소에는 적합하지만, 금융회사의 보안 아키텍처와 근본적으로 충돌한다.**
>

### 1.3 CODEVASP — 금융회사가 직접 쓸 수 없는 이유

CODEVASP는 Enclave 설치는 필요 없지만:

| 문제 | 상세 |
| --- | --- |
| **HTTPS 외부 통신** | `trapi.codevasp.com`에 HTTPS API 호출 필요 → 망분리 위반 |

### 1.4 Sumsub, Notabene 등 해외 TR허브 — 금융회사가 쓸 수 없는 이유

| 문제 | 상세 |
| --- | --- |
| **PII 평문 전달 필수** | API에 이름·생년월일·여권번호를 **평문 JSON**으로 보내야 함 |
| **해외 서버로 개인정보 전송** | Sumsub 서버(해외)에 고객 개인정보가 평문으로 전달 → 개인정보 국외이전 이슈 |
| **HTTPS 외부 통신** | 동일한 망분리 위반 문제 |

## Part 2. Transight(TTR)이 브릿지가 되어야 하는 이유

### 2.1 우리의 인프라 강점

| 인프라 | 설명 | 의미 |
| --- | --- | --- |
| **전자금융보조업자** | 금융위 등록 사업자 | 금융회사와 합법적으로 전자금융 관련 업무 위탁·수탁 가능 |
| **VAN사** | 부가가치통신사업자 | 금융 데이터 중계 전문 사업자로서의 법적 지위 |
| **하나은행 전용선** | 이미 구축된 전용 통신선 | 망분리 요건 100% 충족하는 통신 채널이 이미 존재 |
| **ISMS 인증** | 정보보호관리체계 인증 | 금융회사가 요구하는 보안 기준 충족 |
| **mTLS / VPN** | 금융회사급 보안 통신 | 전용선 외 추가 옵션으로 금융회사 요건 대응 가능 |

### 2.2 TTR 브릿지 구조

```
┌──────────────┐   전용선/VPN    ┌──────────────┐   HTTPS    ┌──────────────┐
│   금융회사    │ ◄════════════► │     TTR      │ ◄────────► │  CODEVASP    │
│  (하나은행)   │   망분리 준수  │   (브릿지)    │            │  (빗썸 등)   │
│              │               │              │            ├──────────────┤
│  ·업무서버    │               │  ·전자금융    │            │  VerifyVASP  │
│  ·KYC DB     │               │    보조업자   │            │  (업비트 등) │
│              │               │  ·VAN사       │            ├──────────────┤
│              │               │  ·ISMS 인증   │            │     GTR      │
│              │               │              │            │ (Binance 등) │
└──────────────┘               └──────────────┘            └──────────────┘
     내부망                      금융 전용 인프라                인터넷
```

**금융회사는 TTR과만 전용선으로 통신.** 인터넷에 한번도 나가지 않음.
**TTR이 기존 TR 솔루션들과의 프로토콜 변환·중계를 대행.**

## Part 3. 거래소 설득 논리 (빗썸·업비트)

### 3.1 핵심 메시지

> **"지금 쓰고 있는 CODEVASP/VerifyVASP만으로는 스테이블코인 시대에 금융기관과 Travel Rule을 할 수 없다. 우리가 그 브릿지를 해줄 수 있으니, 지금 연동해두면 규제 시행 시 바로 대응 가능하다."**
> 

### 3.2 설득 포인트

### ① 스테이블코인 유통 시 금융기관이 반드시 참여한다

```
현재:   거래소 ↔ 거래소  (CODEVASP/VV로 충분)
미래:   거래소 ↔ 은행    (기존 솔루션으로 불가!)
        거래소 ↔ 증권사
        거래소 ↔ 보험사
```

스테이블코인 관련 법안이 시행되면 은행·증권사가 디지털 자산 유통에 참여한다. 이때 **금융기관 ↔ 거래소 간 Travel Rule 의무**가 발생하는데, 금융기관은 망분리 때문에 기존 솔루션을 직접 쓸 수 없다.

### ② 거래소에 추가 비용·부담이 거의 없다

- **빗썸** (CODEVASP 사용): CODEVASP에 NaCl Box로 암호화된 payload를 보내는 **동일한 구조 그대로** TTR에도 연동하면 됨. 프로토콜이 같으니 개발 부담 최소화.
- **업비트** (VerifyVASP 사용): TTR Adapter 연동 한 번이면 금융기관 전체에 대한 TR 대응 완료.
- 기존 솔루션을 대체하는 게 아니라 **병행 운영**. 거래소 간은 기존 솔루션, 금융기관 간은 TTR.

### ③ 선점 효과

규제 시행 후 연동하면 늦다. **지금 연동해두면**:

- 규제 시행 즉시 금융기관과 TR 가능 (경쟁사 대비 우위)
- 스테이블코인 유통 파트너십에서 유리한 포지션

### 3.3 빗썸 특화 논리

> "빗썸은 CODEVASP를 쓰고 있다. CODEVASP의 NaCl Box 암호화 + IVMS101 구조를 **그대로** TTR에도 쓸 수 있다. 사실상 엔드포인트만 하나 추가하는 수준이다. 그러면 빗썸이 금융회사(하나은행 등)와 Travel Rule을 하는 건 우리가 책임진다."
> 

```
빗썸 → CODEVASP (기존 유지, 거래소 간)
빗썸 → TTR      (추가, 금융기관 간) ← 동일한 NaCl Box 프로토콜
```

### 3.4 업비트 특화 논리

> "업비트는 VerifyVASP를 쓰고 있지만, VV의 Enclave 방식은 금융회사에서 절대 도입 불가다. Docker 컨테이너를 은행 내부망에 심는 걸 은행 보안팀이 허용할 리 없다. TTR Adapter 하나 꽂으면 금융기관 전체 커버 가능하다."
>

## Part 4. 하나은행 설득 논리

### 4.1 핵심 메시지

> **"KYT와 Travel Rule을 한번에, 금융회사가 실제 쓸 수 있는 환경에서 먼저 구축하면 스테이블코인 시장의 표준이 될 수 있다. Transight와 안 하면 시간만 지연된다."**
> 

### 4.2 설득 포인트

### ① 현재: 금융회사가 TR을 할 방법이 없다

```
하나은행이 직접 하려면?
  → CODEVASP 가입?  → 망분리 위반 (HTTPS)
  → VerifyVASP?     → Enclave Docker 설치 불가
  → Sumsub?         → PII 평문으로 해외 전송 불가
  → GTR?            → 망분리 위반 (HTTPS)
  → 자체 개발?      → 2~3년 + 수십억 비용
```

**현실적으로 Transight(TTR)을 통하지 않으면 단기간 내 금융회사 환경에서 TR을 구현할 방법이 없다.**

### ② Transight는 이미 인프라가 있다

- **하나은행 전용선**: 이미 구축 완료. 추가 인프라 투자 불필요.
- **전자금융보조업자**: 금융위 등록 사업자로서 금융 데이터 처리 합법적 위탁 가능.
- **VAN사**: 금융 데이터 중계 사업자로서의 검증된 보안 체계.
- **KYT 이미 개발**: 트랜잭션 모니터링 + Travel Rule을 **하나의 API**로 제공 가능.

### ③ 시장 선점 = First Mover Advantage

```
시나리오 A: 하나은행이 TTR과 먼저 구축
  → 스테이블코인 법안 시행 즉시 TR 가능
  → 빗썸·업비트와 바로 연결 (TTR이 브릿지)
  → "금융권 최초 TR 구현 은행" → 시장 표준 주도

시나리오 B: 하나은행이 지금 안 함
  → 법안 시행 후 뒤늦게 솔루션 검토 시작
  → 12~18개월 지연 (보안 심사, 개발, 테스트)
  → 경쟁 은행에 선점당할 리스크
```

### ④ KYT + TR 원스톱

```
기존 방식:
  은행 → KYT 솔루션 (Chainalysis 등) + TR 솔루션 (???) → 별도 연동 2건

TTR 방식:
  은행 → TTR 하나로 KYT + TR 동시 해결 → 연동 1건
         │
         ├── Atomic KYT Gate: 트랜잭션 위험도 자동 분석
         └── Travel Rule: 거래소·금융기관 간 PII 교환
```

비용 절감 + 연동 부담 감소 + 일관된 컴플라이언스 체계.

## Part 5. 한 장 요약

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   문제: 금융기관은 기존 TR 솔루션을 쓸 수 없다          │
│   ├── VV Enclave Docker → 보안 불가                     │
│   ├── CODEVASP HTTPS → 망분리 위반                      │
│   ├── Sumsub 평문 PII → 개인정보 이슈                   │
│   └── GTR HTTPS → 망분리 위반                           │
│                                                         │
│   해결: TTR이 브릿지가 된다                             │
│   ├── 전용선/VPN으로 금융회사와 통신 (망분리 준수)       │
│   ├── HTTPS로 기존 TR 솔루션과 통신 (프로토콜 변환)     │
│   ├── E2E 암호화 유지 (Curve25519/NaCl)                 │
│   └── KYT + TR 원스톱 제공                              │
│                                                         │
│   거래소에게: "우리 꽂아두면 금융기관 대응 끝"           │
│   은행에게:   "우리 쓰면 시장 선점 가능, 안 쓰면 지연"  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```
